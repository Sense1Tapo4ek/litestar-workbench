# Спецификация написания уроков — S1T Litestar Workbench

---

## Аудитория

Python-разработчик, знакомый с FastAPI или Django. Понимает async/await, REST, Pydantic, аннотации типов. **Не нужно** объяснять что такое HTTP, JSON, декораторы. **Нужно** показывать: чем Litestar отличается, что он умеет, где его API специфичен.

---

## Язык и терминология

- Теория пишется на **русском**
- Технические термины **не переводятся**: `handler`, `route`, `middleware`, `dependency`, `guard`, `DTOFactory`, `Provide`, `State`, `lifespan`
- Переменные, функции, классы в примерах — **на английском** (`item_id`, `get_items`, `Item`)
- Названия HTTP-методов и статус-кодов — **заглавными без кавычек**: GET, POST, 201, 404

---

## Тон

Технический и лаконичный. Только факты — без вводных слов, без «как мы знаем», без мотивирующих фраз.

**Так нельзя:**
> Сегодня мы познакомимся с замечательной возможностью Litestar — dependency injection, которая очень помогает в реальных проектах!

**Так нужно:**
> Litestar позволяет объявить зависимость один раз и автоматически подставить её в любой хендлер.

## Структура урока по режиму

### `tutorial` — новая концепция

1. **Заголовок** — название концепции
2. **Суть** — 1–3 предложения: что это, зачем
3. **Пример кода** — полный рабочий сниппет
4. **Разбор** — таблица или список ключевых деталей
5. **Edge cases** — отдельный блок в конце (см. ниже)

### `challenge` — закрепление

1. **Задание** — что нужно реализовать (глагол + существительное)
2. **Таблица эндпоинтов** — метод, путь, статус, краткое описание
3. **Подсказка** — только если без неё задание нечестное

### `bugfix` — поиск ошибок

1. **Контекст** — что должен делать код
2. **Симптом** — какие тесты падают и почему это странно
3. Код — с ошибками, **без указания где именно**

## Объём `lesson.md`

| Режим | Объём |
|-------|-------|
| `tutorial` | 150–400 строк |
| `challenge` | 60–120 строк |
| `bugfix` | 40–80 строк |

Если тема требует больше 400 строк — разбить на два урока.

## Примеры кода в теории

**Короткие фокусированные сниппеты** — показывают только ключевую идею, без импортов и `app = Litestar(...)`. Полный рабочий код всегда доступен в редакторе слева.

```python
@get("/items/{item_id:int}")
async def get_one_item(item_id: int) -> Item:
    item = get_item(item_id)
    if item is None:
        raise NotFoundException(f"Item {item_id} not found")
    return item
```

Если нужно показать импорт конкретного символа — можно вынести одной строкой перед сниппетом:

```python
from litestar.exceptions import NotFoundException
```

Никогда не включать `app = Litestar(...)` в сниппеты теории — это уже видно в редакторе.

`...` допустимо только когда явно показываем структуру (сигнатуру функции), не поведение.

## Edge cases

Каждый tutorial-урок заканчивается разделом **«На что обратить внимание»**. Содержит 2–4 пункта — неочевидные поведения, типичные ошибки, ограничения.

```markdown
## На что обратить внимание

- `status_code` на уровне декоратора перекрывает код из `Response`
- Если тип возвращаемого значения не задан, Litestar не сериализует ответ автоматически
- `NotFoundException` генерирует тело `{"detail": "..."}` — не пустой 404
```

Ошибки и антипаттерны — только в bugfix-уроках, не в tutorial.

## Граница урока

**Одна концепция — один урок.**

Признаки того, что тема требует разбивки:
- Теория занимает больше 400 строк
- Сценарий содержит больше 8 шагов
- В `lesson.py` больше 80 строк основной логики

Примеры правильной разбивки:

| Неправильно | Правильно |
|-------------|-----------|
| «Параметры запроса» (path + query + headers + body) | «Path parameters», «Query parameters», «Request body» — три урока |
| «Middleware и Guards» | «Middleware», «Guards» — два урока |

## Сценарий (scenario.yaml)

### Именование шагов

Имя шага — **глагол + объект** на русском, без технических деталей пути:

| Плохо | Хорошо |
|-------|--------|
| `GET /items 200` | `Получить список` |
| `Test create endpoint` | `Создать элемент` |
| `Step 3` | `Удалить несуществующий` |

### Поле `description`

Всегда заполнять — объяснить **поведение**, а не повторять метод и путь:

```yaml
- name: "Пустой список"
  method: GET
  path: /items
  expect_status: 200
  description: "Возвращает [] — не 404 и не null"

- name: "Удалить несуществующий"
  method: DELETE
  path: /items/9999
  expect_status: 404
  description: "Повторное удаление возвращает 404, не 204"
```

### Порядок шагов

Стандартная последовательность для CRUD:
1. Пустое состояние (список пустой)
2. Создать
3. Прочитать по ID
4. Обновить
5. Удалить
6. Подтвердить удаление (повторный GET → 404)
7. Граничные случаи: несуществующий ID, невалидные данные

### Переменные

Использовать `save_as` когда следующий шаг зависит от ID созданного объекта. Имя переменной — имя ресурса в единственном числе: `item`, `user`, `order`.

```yaml
- name: "Создать"
  method: POST
  path: /items
  body: {name: "Книга", price: 10.0}
  expect_status: 201
  save_as: item

- name: "Прочитать"
  method: GET
  path: "/items/${item.id}"
  expect_status: 200
```

## `lesson.py` — code style

### Именование

| Элемент | Конвенция | Пример |
|---------|-----------|--------|
| Хендлеры | `<глагол>_<ресурс>` | `get_items`, `create_item`, `delete_item` |
| Модели | PascalCase | `Item`, `UserProfile` |
| In-memory хранилище | `_store` | `_store: dict[int, Item] = {}` |
| Счётчик ID | `_next_id` | `_next_id = 1` |
| Вспомогательные функции | snake_case | `find_item`, `validate_price` |

### Обязательные правила

- Нет модульного docstring в `lesson.py`
- Все хендлеры — `async def` с полными аннотациями типов
- Объект приложения — ровно `app = Litestar(...)`
- Нет `print()` — для отладки не нужен

### Импорты

Группировать в порядке: stdlib → litestar → local

```python
from __future__ import annotations

from litestar import Litestar, delete, get, patch, post, put
from litestar.exceptions import NotFoundException
from litestar.status_codes import HTTP_201_CREATED, HTTP_204_NO_CONTENT

from models import Item, create_item, delete_item, get_item, list_items, update_item
```

## Challenge-урок: структура `lesson.py`

Частичный код с `# TODO`-комментариями. Стартовая точка — не пустой файл, а скелет:

```python
from litestar import Litestar, get, post
from litestar.status_codes import HTTP_201_CREATED

from models import Item, create_item, list_items


# TODO: реализуй GET /items — возвращает list[Item]


# TODO: реализуй POST /items — создаёт Item, возвращает 201


app = Litestar(route_handlers=[])
#                               ↑ добавь хендлеры сюда
```

Правила:
- `# TODO:` всегда с двоеточием и пробелом
- Описание в `# TODO` — конкретное: **что** реализовать, **какой** статус, **что** вернуть
- Импорты уже написаны — студент пишет только хендлеры
- `models.py` и все вспомогательные функции — **готовы**
- `app = Litestar(route_handlers=[])` — заготовка есть, студент добавляет хендлеры

## Bugfix-урок: структура ошибок

1–3 ошибки разного типа в одном уроке. Типы ошибок:

| Тип | Пример |
|-----|--------|
| Неправильный status code | `201` вместо `204` на DELETE |
| Отсутствующая аннотация | нет `-> Item` → Litestar не сериализует |
| Неверный путь параметра | `{item_id}` без типа вместо `{item_id:int}` |
| Логическая ошибка | `_store.get(item_id - 1)` вместо `_store.get(item_id)` |
| Пропущенный хендлер в `app` | хендлер написан, но не добавлен в `route_handlers` |

Правила:
- Ошибки не помечаются комментариями
- `lesson.md` описывает **симптом**, а не место ошибки: «сервер возвращает 404 на DELETE /items/1, хотя элемент существует»
- Тесты в `scenario.yaml` проверяют корректное поведение — они падают до исправления

## Ссылки из теории в код

Использовать `code:`-ссылки когда теория ссылается на конкретное место в `lesson.py`:

```markdown
Хендлер [get_items](code:lesson.py#L8) возвращает `list[Item]`.

Модель данных описана в [models.py](code:models.py).
```

Не ссылаться на строки, которые студент будет изменять (в challenge/bugfix) — номер строки устареет.

## Чек-лист перед публикацией урока

**Файлы:**
- [ ] `lesson.yaml` — заполнены `title`, `order`, указан `mode`
- [ ] `lesson.py` — нет docstring, есть `app = Litestar(...)`, все типы проставлены
- [ ] `lesson.md` — написан в соответствии с режимом урока
- [ ] `scenario.yaml` — все шаги именованы, заполнен `description`

**Качество:**
- [ ] Теория вписывается в лимит строк для режима
- [ ] Все примеры кода в теории — полные и запускаемые
- [ ] Последний раздел tutorial-урока — «На что обратить внимание»
- [ ] Шаги сценария покрывают happy path + минимум 1–2 граничных случая
- [ ] Урок посвящён одной концепции

**Проверка запуском:**
- [ ] `Start` — сервер запускается без ошибок
- [ ] Все шаги сценария проходят на готовом коде
- [ ] Для challenge/bugfix: шаги **не** проходят на стартовом коде
